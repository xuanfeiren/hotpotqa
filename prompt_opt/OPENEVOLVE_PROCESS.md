# OpenEvolve Optimization Process for HotpotQA

This document describes how the OpenEvolve prompt optimization is configured and how it works, step by step.

---

## Overview

OpenEvolve uses **evolutionary search** to discover better prompts. It maintains a population of prompt candidates, generates new candidates by mutating existing ones via an LLM, evaluates them on HotpotQA tasks, and keeps the best ones.

---

## Current Configuration

| Parameter | Value | Notes |
|---|---|---|
| Max iterations | 20 | Each iteration = 1 new prompt proposal |
| Population size | 500 | Max prompts in the database |
| Num islands | 3 | Separate sub-populations for diversity |
| Stage 1 eval samples | 20 tasks | Quick screening (first 20 of the 100) |
| Stage 2 eval samples | 100 tasks | Full evaluation — same 100 tasks as Trace |
| Cascade threshold | 0.5 | Stage 1 must score ≥ 50% to proceed to stage 2 |
| Evolution mode | Full rewrite | LLM writes the entire new prompt |
| Optimizer LLM | `gemini-2.5-flash-lite` | Same model as used for evaluation |

### Task Selection

**Both Trace and OpenEvolve use the same 100 tasks** — the first 100 items from HotpotQA validation set (distractor setting), loaded via `create_dataset(n=100)`. No random sampling; same fixed order every time.

The cascade stage 1 uses the **first 20 of those 100 tasks** as a cheap pre-filter. If a prompt scores below 50% on tasks 0–19, it's rejected without running the full 100. Stage 2 always evaluates all 100.

---

## Step-by-Step Process

### 1. Initialization

1. Load the initial prompt from `openevolve_initial_prompt.txt`
2. Evaluate it through the cascade:
   - **Stage 1**: Run on 20 tasks → get accuracy
   - If accuracy ≥ 0.5 (cascade threshold), proceed to **Stage 2**: Run on 100 tasks → get final accuracy
3. Store the initial prompt + its score in the population database (island 0)

### 2. Each Evolution Iteration (20 total)

#### 2a. Select a parent prompt

The database uses **island-based evolution** with 3 islands. A parent is sampled from the current island using a probabilistic strategy:

- **30% exploration**: Random selection from the island
- **60% exploitation**: Select from the archive (elite prompts)
- **10% weighted**: Fitness-weighted sampling

#### 2b. Build the evolution prompt

The system constructs an LLM prompt for the optimizer containing:

- The parent prompt (current candidate being evolved)
- Its evaluation metrics (accuracy score)
- **Artifacts from the last evaluation** — per-question feedback showing:
  - Question text
  - Expected answer
  - Model output
  - Whether it was correct or incorrect
- Error analysis summary (which questions it got wrong and why)
- Evolution history (top programs from the island, previous attempts)
- System instructions for prompt engineering

**This is the same information Trace sees**: question, context, expected answer, and model output.

#### 2c. Generate a new prompt candidate (1 proposal = 1 LLM call)

The optimizer LLM (`gemini-2.5-flash-lite`) receives the evolution prompt and generates a **full rewrite** of the prompt. This is parsed to extract the new prompt text.

#### 2d. Evaluate the new prompt (cascade)

- **Stage 1** (20 tasks, tasks 0–19): If accuracy < 50%, reject → skip to next iteration
- **Stage 2** (100 tasks, tasks 0–99): Full evaluation → record final accuracy

#### 2e. Store the result

The new prompt + metrics + artifacts are stored in the database:
- **MAP-Elites grid**: Prompts are placed in cells based on `prompt_length` feature dimension
- If the new prompt is better than the existing prompt in its cell, it replaces it
- The global best prompt is tracked separately and never lost

### 3. Island Management

- Every 5 iterations (`migration_interval`), the best prompts migrate between islands (10% migration rate)
- This prevents convergence to a single local optimum

---

## Logging

Each evaluation step prints:

- **Current accuracy**: Accuracy on the current stage's tasks
- **Best accuracy so far**: Best score across all prompts evaluated
- **Total metric calls**: Cumulative count of individual task evaluations (1 task = 1 metric call)
- **Total proposals**: Number of new prompt candidates generated by the optimizer LLM (unique prompts - 1)
- **Best prompt so far**: The text of the highest-scoring prompt

State is persisted to a JSON file to survive across worker processes (OpenEvolve uses `ProcessPoolExecutor`).

## Key Differences from Trace

| Aspect | Trace | OpenEvolve |
|---|---|---|
| **Eval tasks** | First 100 (fixed) | Same first 100 (fixed), with 20-task cascade pre-filter |
| **Search strategy** | Gradient-like (LLM feedback propagation) | Population-based evolutionary search (MAP-Elites + islands) |
| **Feedback to optimizer** | Per-question results via computational graph | Per-question results via text artifacts in the LLM prompt |
| **Prompt generation** | Targeted parameter updates via OptoPrime | Full rewrite of the entire prompt |
| **Diversity mechanism** | Epsilon-net priority search | 3 islands + MAP-Elites feature grid |
| **Optimizer per step** | 1 LLM call (to OptoPrime) | 1 LLM call (to generate rewrite) |
| **Metric calls per step** | 1 (single task) | 20 (stage 1) + 100 (stage 2 if passed) |
